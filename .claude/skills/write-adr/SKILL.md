# ADR 作成ワークフロー

マージされた PR を分析し、ADR に値する設計判断が含まれる場合のみ docs/adr/ に ADR を追加する PR を作成する。

## 前提

- 引数: PR 番号（必須）。`$ARGUMENTS` に含まれる。
- ADR のテンプレートと運用ルールは docs/adr/README.md と docs/adr/0000-template.md を参照。

## 手順

### 1. PR の分析

1. `gh pr view <番号> --comments` と `gh pr diff <番号>` で変更内容・議論・関連 Issue を読む。
2. **ADR に値するか判断する。** 以下のいずれかに該当する場合のみ作成する:
   - アーキテクチャ・構成の変更（レイヤ構造、状態管理方式、ディレクトリ設計など）
   - 技術選定（ライブラリ・フレームワーク・ツールの採用/置換/廃止）
   - 横断的な規約の導入・変更（テスト方針、エラーハンドリング方針など）
   - セキュリティ・パフォーマンスに関わるトレードオフの選択
3. 該当しない場合（バグ修正、軽微な機能追加、依存バージョン更新など）は
   「ADR 不要」と判断した理由を出力して**何も作成せず終了する**。判断に迷う場合も作成しない。

### 2. 既存 ADR の確認

- docs/adr/ の既存 ADR を確認し、矛盾・置換関係がないか確認する。
- 既存 ADR を置き換える決定の場合は、新 ADR に「置換: ADR-XXXX」を明記し、
  旧 ADR の Status を Superseded に更新する。

### 3. ADR の作成

1. docs/adr/0000-template.md をベースに、次の連番で `docs/adr/NNNN-<英語ケバブケース>.md` を作成する。
2. 内容は PR の議論から再構成する。**事後の正当化ではなく、当時の選択肢と判断理由を記録する**:
   - Context: 何が課題だったか
   - Decision: 何を選んだか
   - Alternatives: 検討した代替案と却下理由（PR 議論から読み取れる範囲で）
   - Consequences: この決定で生じる利点・制約・フォローアップ
3. 元の PR と Issue への参照リンクを含める。

### 4. PR の作成

1. ブランチ `claude/adr-<PR番号>` を作成し、ADR ファイルをコミットして push する。
2. `gh pr create` で PR を作成する。タイトル: `docs(adr): <決定の要約>`。
3. 本文に元 PR への参照と、ADR 作成判断の根拠を記載する。

## 完了条件

- ADR に値する場合: ADR ファイルを含む PR が作成されている
- 値しない場合: 理由を出力して終了している（ファイル変更なし）
