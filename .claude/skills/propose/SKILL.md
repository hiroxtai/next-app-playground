# 改善提案ワークフロー

コードベースを調査して価値のある改善提案を複数提示し、ユーザーの選択を受けて
実装可能なタスク（Issue）まで落とし込む。
プロジェクト固有の文脈は CLAUDE.md と docs/adr/ を参照すること。

## 手順

### 1. 現状調査

以下を並行して調査する:

- コードベースの構成と最近のコミット履歴（`git log --oneline -30`）
- オープン中の Issue / PR（`gh issue list`, `gh pr list`）
- 過去の ADR（docs/adr/）— 既に却下された案を再提案しない
- プロジェクトの目的に照らした未実装領域（このプロジェクトでは学習カタログの未収録パターンなど）
- 技術的負債・テストカバレッジの穴・依存関係の古さ

### 2. 提案の提示

3〜5件の提案を提示する。各提案に含めるもの:

- **タイトル**と一行説明
- **価値**: なぜやる価値があるか（プロジェクトの目的との関連）
- **規模感**: 小（〜半日）/ 中（〜数日）/ 大（要分割）
- **リスク・前提**: 破壊的変更の有無、新規依存の有無

提示したらユーザーの選択を待つ。**勝手に実装を始めない。**

### 3. 選択された提案の構造化

規模に応じて進め方を変える:

- **小規模**: 要件と受け入れ条件をまとめ、そのまま 1 つの Issue として起票する。
- **中〜大規模**: Spec Kit のフローで構造化する。
  1. `/speckit-specify` で仕様を作成
  2. 曖昧さが残れば `/speckit-clarify` で解消（ユーザーに質問する）
  3. `/speckit-plan` で技術設計
  4. `/speckit-tasks` でタスク分割

### 4. Issue 起票

1. 分割された各タスクを `gh issue create` で起票する。Issue 本文に含めるもの:
   - 背景（仕様・プランへの参照）
   - 受け入れ条件（テスト可能な形で）
   - 依存する他の Issue（あれば番号で明記）
2. ユーザーに確認の上、自動処理してよいタスクに `claude:auto` ラベルを付ける
   （ラベルを付けると resolve-issue パイプラインが自動で発火する）。
3. 起票した Issue の一覧をユーザーに報告する。

## 完了条件

- ユーザーが選んだ提案が、受け入れ条件付きの Issue 群になっている
- 自動処理の承認（ラベル付け）はユーザーの判断を経ている
