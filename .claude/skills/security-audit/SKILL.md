# セキュリティ監査ワークフロー

プロジェクトを点検し、対応が必要な問題を「調査結果と対応方針付きの Issue」として起票する。
このスキルは**調査と起票のみ**を行う。コードの変更は行わない（修正は resolve-issue パイプラインが担当する）。

## 手順

### 1. 既存状況の確認

重複起票を避けるため、先に以下を確認する:

- オープン中の `security` ラベル付き Issue（`gh issue list --label security`）
- オープン中の Dependabot PR / アラート（`gh pr list --author "app/dependabot"`）

### 2. 依存関係の監査

1. パッケージマネージャの監査コマンドを実行する（pnpm なら `pnpm audit --json`）。
2. 検出された脆弱性ごとに、severity・影響パッケージ・修正バージョンの有無を整理する。
3. Dependabot が既に PR を出しているものは除外する。

### 3. コードの点検

OWASP Top 10 を観点に、プロジェクトのコードを点検する。特に:

- シークレット・API キーのハードコード（環境変数の扱いを含む）
- 外部入力の検証不足（API ルート、フォーム、URL パラメータ）
- XSS の可能性（`dangerouslySetInnerHTML` など）
- 認証・認可の欠落（公開すべきでないエンドポイント）
- AI 連携部分のプロンプトインジェクション耐性（外部入力をそのまま LLM に渡していないか）
- 依存ライブラリの非推奨・メンテナンス停止

### 4. CI / ワークフローの点検

- GitHub Actions の permissions が最小権限になっているか
- サードパーティ Action のバージョン固定（タグ/SHA）がされているか
- シークレットの扱いに問題がないか

### 5. 結果の起票

対応が必要な問題ごとに `gh issue create` で Issue を起票する。本文に含めるもの:

- 問題の説明と発見場所（ファイルパス・依存名）
- 重大度（Critical / High / Medium / Low）と判断理由
- 推奨する対応方針（具体的に）
- 参考リンク（CVE、Advisory など）

ラベル付けのルール:

- 全件に `security` ラベルを付ける
- **`claude:auto` は付けない。** 自動修正パイプラインに流すかどうかは人間がラベル付けで判断する
  （Critical の場合は Issue タイトルに [Critical] を付けて目立たせる）

### 6. サマリ報告

問題が 0 件の場合も含め、監査結果のサマリを出力する（定期実行ログとして残る）:

- 点検した範囲
- 起票した Issue の一覧（番号・タイトル・重大度）
- 既存 Issue / Dependabot でカバー済みのためスキップした項目

## 完了条件

- 監査が完了し、対応が必要な問題が全て Issue 化されている
- コードの変更を一切行っていない
