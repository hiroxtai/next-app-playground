自律型コーディングエージェント開発環境
— 調査と設計

調査日: 2026年6月10日 ／ 対象リポジトリ: next-app-playground ／ ブランチ: feature/orchestrator ／ 姉妹ドキュメント: GitHub Copilot 版

結論（TL;DR）
2026年6月時点のベストプラクティスは、「賢い単一のオーケストレーターエージェント」を自作することではなく、「GitHub をイベントバス兼状態管理として使うイベント駆動パイプライン」を組むこと。 LLM に任せるのは各ステージ内の判断（コードを書く・レビューする・提案する）だけに絞り、ステージ間の遷移と品質ゲートは GitHub Actions・CI・ブランチ保護・フックといった決定論的な仕組みで制御する。 汎用性（どのプロジェクトでも使える性質）は、手順を「スキル」に、プロジェクト固有知識を「CLAUDE.md」に分離することで確保する。

1. 背景と要件

このリポジトリ（Next.js 16 学習プレイグラウンド）で、コーディングエージェントができる限り自律的に開発を進める環境を構築したい。具体的には次の3つのフローを、人間の関与を最小化しながら実現する。

  1. Issue 駆動: Issue が登録されたら、問題の理解 → 解決策の検討 → 実装 → テスト → PR 作成 → レビュー → マージ → ADR 作成までを自動で行う。
  2. 提案駆動: ユーザーが「何か新しい提案はない？」と尋ねたら複数の提案を行い、指示があれば要件検討 → 設計 → プラン → タスク分割 → 実装 → マージ → ADR まで進める。
  3. 定期スキャン駆動: 毎日セキュリティ関連の問題を調査し、対応すべき問題があれば対応方法の調査 → プラン → 実装 → マージ → ADR まで進める。

加えて、特定プロジェクトに依存しない汎用的な仕組みであることを重視する。

2. なぜ「単一オーケストレーター」方式を採らないのか

「強力なオーケストレーターエージェントがサブエージェントを指揮する」構成は広く発信されているが、調査の結果、1セッション内の並列化には有効でも、システム全体の制御を1つの LLM に委ねるのは現時点ではアンチパターンに近いと判断した。理由は次の3点。

  1. 信頼性は「賢さ」ではなく「構造」から生まれる。 2026年の自律エージェント成功事例の分析では、成功要因はモデル性能ではなく「テストインフラへの投資」「速いフィードバックループ」「明確なチェックポイント」で一致している。LLM オーケストレーターは実行のたびに判断が揺れ、失敗時のデバッグが困難になる。
  2. 状態管理が破綻する。 長時間動くオーケストレーターはコンテキストが肥大化し、途中で停止すると全状態を失う。状態を GitHub（Issue・PR・ラベル・コメント）に持たせれば、各エージェント実行はステートレスになり、失敗したステージだけを再実行できる。
  3. スペック駆動開発の知見。 GitHub Spec Kit の実践報告では、「全部お任せ」型では10〜20分のタスクが委任の限界だったのに対し、仕様 → 計画 → タスク分割のチェックポイント構造を入れることで、数時間規模の機能開発を安全に委任できるようになったとされる。自律度を上げる鍵はエージェントの賢さではなくパイプラインの構造化である。

3. 推奨アーキテクチャ（4層構成）

下図の4層で構成する。下の層ほど「信頼の土台」となる。

トリガー層 決定論的

人間の関与を GitHub イベントに置き換える。issues: [labeled]、pull_request、schedule: cron、および @claude メンション。ローカルの対話セッションも「提案駆動」フローの入口としてここに含まれる。

実行層 claude-code-action@v1

anthropics/claude-code-action は GitHub Actions ランナー上でフル機能の Claude Code を実行する公式 Action。@claude メンションへの応答（対話モード）と prompt 指定による無人実行（自動化モード）を自動判別する。ローカルの Claude Code と同じ CLAUDE.md・同じスキルを読むため、ローカルで磨いたワークフローがそのまま CI で動く。

共有定義層 — 汎用性の鍵

手順（Issue 解決・ADR 作成・セキュリティ監査など）はスキルとして書き、プロジェクト固有の知識（規約・コマンド・構成）は CLAUDE.md に書く。スキルは CLAUDE.md を参照する前提でプロジェクト非依存に書く。この分離により、スキル群とワークフロー YAML のセットを任意のリポジトリへ数分で導入できる（プラグイン化・テンプレートリポジトリ化も可能）。

ガードレール層 決定論的

  • CI: lint / type-check / test を required status check に。エージェントの成果物も人間のコードと同じ検証を通す。
  • フック: 「テストが通るまで停止しない」「生成ファイルの編集禁止」などをプロンプトではなくコードで強制（フックはハルシネーションしない）。
  • ブランチ保護: main への直接 push 禁止、PR レビュー必須。

5. 3つのユースケースの実現方法

3つのフローはすべて「ラベル付き Issue」という共通フォーマットに正規化して、単一の実装パイプラインに合流させる。これがイベント駆動設計の最大の利点で、仕組みが最小で済む。

A. Issue 駆動（自動解決）

1つの巨大ワークフローではなく、イベントで連鎖する小さなワークフローの列として組む。各段が独立しているため、どこで失敗しても再実行・人間介入が容易。

  1. claude:auto ラベルが付いた Issue に対して /resolve-issue スキルが発火（ラベルゲート）
  2. スキルが調査 → 解決方針を Issue にコメント → 実装 → テスト合格まで修正 → PR 作成
  3. PR イベントで自動レビューワークフローが発火
  4. CI（lint / type-check / test）が required check として検証
  5. マージ後、ADR ワークフローが発火（ADR に値する決定かはスキルが判断）

B. 提案駆動

入口はローカルの Claude Code セッション（提案の取捨選択は人間の判断そのもののため）。

  1. /propose スキル: コードベース・Issue 一覧・未実装パターンを調査し、複数の提案を提示
  2. ユーザーが選択したら Spec Kit の6段階（Constitution → Specify → Clarify → Plan → Tasks → Implement）で構造化
  3. タスク分割後、各タスクを claude:auto ラベル付き Issue として起票 → パイプライン A に合流

C. 定期セキュリティ監査

  1. cron（毎日）で /security-audit スキルが発火
  2. 依存関係の既知脆弱性・コードのセキュリティパターン・GitHub Advisory を点検
  3. 対応が必要なら調査結果・対応方針付きの Issue を起票 → パイプライン A に合流

Dependabot や CodeQL は決定論的なスキャナとして併用し、エージェントは「判断が必要な調査」に専念させる。

6. セキュリティと自律度の段階制

完全無人のマージは公式に非推奨。理由は品質ではなくセキュリティ。Issue 本文・コメントは「信頼できない入力」であり、隠し命令（HTML コメント・不可視文字など）によるプロンプトインジェクション攻撃が2026年に実際に多発している。OWASP Agentic Top 10 でも「Agent Goal Hijacking」が1位。claude-code-action の公式セキュリティドキュメントも自動マージを明確に非推奨としている。

その他のセキュリティ設計:

  • ワークフローの permissions は最小権限（ジョブごとに必要なものだけ）
  • API キーは GitHub Secrets（ANTHROPIC_API_KEY）でのみ管理
  • ラベルゲート自体がインジェクション対策になる（外部の人が Issue を立てるだけでは発火しない。人間がラベルを付けるときに本文を一読する）
  • 定期監査ジョブは contents: read のみ（Issue 起票だけ許可、コード変更はパイプライン A 経由）

7. このリポジトリでの実装構成

.github/workflows/
├── claude.yml                  # @claude メンション応答（対話）
├── claude-resolve-issue.yml    # claude:auto ラベル → 実装 → PR
├── claude-review.yml           # PR 自動レビュー
├── claude-adr.yml              # マージ後の ADR 作成
└── claude-security-scan.yml    # cron 日次セキュリティ監査
.claude/skills/
├── resolve-issue/SKILL.md      # Issue 解決の手順書
├── propose/SKILL.md            # 改善提案の手順書
├── security-audit/SKILL.md     # セキュリティ監査の手順書
└── write-adr/SKILL.md          # ADR 作成の手順書
docs/adr/
├── README.md                   # ADR の運用ルール
├── 0000-template.md            # ADR テンプレート
└── 0001-*.md                   # 個別の ADR
CLAUDE.md                       # 自律パイプラインの節を追記

既存資産との関係:

  • ci.yml（lint / type-check / test / build）はそのまま品質ゲートとして機能
  • dependabot.yml + dependabot-auto-merge.yml は決定論的スキャナ・低リスク自動マージの実例としてレベル3の先行例
  • .claude/skills/speckit-* は提案駆動フローの設計・タスク分割フェーズで使用

8. セットアップ手順

  1. ローカルの Claude Code で /install-github-app を実行（GitHub App のインストールと ANTHROPIC_API_KEY シークレット設定が対話的に完了する）
  2. ラベルを作成: gh label create "claude:auto" --description "エージェントによる自動処理を承認" --color 7F77DD
  3. ブランチ保護を設定: main への直接 push 禁止、required checks に CI のジョブを指定
  4. スキルをローカルで試運転（例: /resolve-issue 123）して手順を磨く — ここが品質の8割を決める
  5. 安定したら、.claude/skills/ + .github/workflows/ をテンプレートリポジトリまたはプラグインとして横展開
運用のコツ: 最初の数週間はレベル2（マージのみ人間）で運用し、PR の品質を観察する。修正なしでマージできた PR の比率が安定して高いカテゴリから、レベル3（条件付き自動マージ）へ拡大する。
