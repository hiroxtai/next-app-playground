GitHub Spec Kit 完全ガイド
仕様駆動開発の進め方

AI コーディングエージェントに「いきなりコードを書かせる」のではなく、 意図 → 仕様 → 設計 → タスク → 実装 という構造化されたプロセスを通すための仕組みが GitHub Spec Kit です。本ガイドでは、このプロジェクトに導入された Spec Kit の具体的な使い方を、 実際のコマンド・プロンプト例・成果物とともに解説します。

対象: このリポジトリ（Copilot 連携済み） コマンド: /speckit.* スラッシュコマンド 成果物: spec.md / plan.md / tasks.md

1Spec Kit とは

GitHub Spec Kit は、Spec-Driven Development（SDD / 仕様駆動開発）を実践するためのツールキットです。 「何を作るか（仕様）」を実行可能なドキュメントとして先に固め、それを入力として AI エージェントに設計・タスク分解・実装を行わせます。

📋 仕様が主役

コードではなく仕様書が「真実の源（source of truth）」。仕様を変えれば実装が追従する、という発想です。

🤖 AI エージェント連携

Copilot / Claude などのエージェントが各フェーズを担当。人間はレビューと意思決定に集中します。

🔁 段階的な成果物

各フェーズが Markdown 成果物を生成し、次フェーズの入力になります。文脈が途切れません。

💡 一言でいうと 「仕様 → 設計 → タスク → 実装」を /speckit.* という一連のスラッシュコマンドで進める、 AI 時代のウォーターフォール × アジャイルの良いとこ取りの開発手法です。

2なぜ仕様駆動開発か

AI に直接「〇〇を作って」と頼むと、前提の取り違え・スコープのブレ・手戻りが頻発します。 Spec Kit はその間に明示的な合意ポイントを挟むことで、品質と再現性を高めます。

いきなり実装を依頼する場合Spec Kit を使う場合
要件の解釈が AI 任せ。意図とズレても気づきにくい spec.md で要件を言語化・合意してから進む
曖昧な点が暗黙のまま実装に紛れ込む /speckit.clarify で曖昧点を質問形式で潰す
技術選定の根拠が残らない plan.md に設計・技術スタックの根拠を記録
大きな塊で生成され、レビューが困難 tasks.md で小さく独立検証可能な単位に分割
プロジェクト規約が無視されがち constitution.md（憲法）で原則を全フェーズに強制

3ディレクトリ構成

specify init でプロジェクトに導入すると、主に次の 3 領域が生成されます。 このリポジトリは Copilot 連携（--integration copilot）で初期化されています。

.specify/ # Spec Kit 本体（設定・テンプレ・スクリプト） ├── memory/ │ └── constitution.md # プロジェクト憲法（不変の原則） ├── templates/ # 各成果物のひな形 │ ├── spec-template.md │ ├── plan-template.md │ ├── tasks-template.md │ └── constitution-template.md ├── scripts/bash/ # 機能ディレクトリ作成等の補助スクリプト ├── workflows/ # コマンドを連結する自動ワークフロー定義 ├── extensions.yml # フック（前後処理）の登録 └── feature.json # 現在作業中の機能ディレクトリを記録 .github/ ├── prompts/ # /speckit.* スラッシュコマンドの入口 │ ├── speckit.specify.prompt.md │ ├── speckit.plan.prompt.md │ └── ... (constitution, tasks, implement 等) └── agents/ # 各コマンドの実体（エージェント定義） ├── speckit.specify.agent.md ├── speckit.plan.agent.md └── ... specs/ # 機能ごとの成果物（フィーチャー単位） └── 001-simple-todo-app/ ├── spec.md # 要件仕様 ├── plan.md # 技術設計 ├── tasks.md # 実装タスク一覧 ├── research.md # 技術調査メモ ├── data-model.md # データモデル ├── contracts/ # API / UI コントラクト └── checklists/ # 品質チェックリスト
🔑 prompts と agents の関係 スラッシュコマンド（例 /speckit.specify）は .github/prompts/ の薄い入口で、 実際の手順・ルールは .github/agents/ のエージェント定義に書かれています。 「コマンドが何をするか」を深く知りたいときは agents 側を読むと分かります。

4開発フロー全体像

SDD の中核は Spec → Plan → Tasks → Implement の 4 フェーズです。 各フェーズが Markdown 成果物を生み、それが次フェーズの入力になります。 前段に「憲法」、間に「明確化」「分析」といった品質ゲートを挟むのが実践的な流れです。

コマンド実行順（パイプライン）

必須フェーズ 任意・推奨フェーズ
0

/speckit.constitution プロジェクト憲法の設定 最初に一度

プロジェクト全体で守る原則（テスト方針・品質基準・技術制約）を定義。以降の全フェーズに影響します。

生成 → .specify/memory/constitution.md
1

/speckit.specify 要件仕様の作成

「何を・なぜ作るか」を自然言語の要望から構造化。ユーザーストーリー・受入基準・成功条件を記述（実装の詳細は書かない）。

入力: 自然言語の要望生成 → spec.md + checklists/requirements.md
2

/speckit.clarify 曖昧点の明確化 推奨

仕様の曖昧な箇所を最大 5 個まで質問。回答を spec.md の「Clarifications」セクションに反映します。

入力: spec.md更新 → spec.md（Clarifications 追記）
3

/speckit.plan 技術設計の作成

仕様をどう実装するかに変換。技術スタック・アーキテクチャ・データモデル・契約を設計し、憲法との整合を確認します。

入力: spec.md + constitution.md生成 → plan.md / research.md / data-model.md / contracts/
4

/speckit.tasks タスク分解

設計を依存順に並んだ実装タスクへ分割。ユーザーストーリー単位でグループ化し、並列実行可能なものに [P] を付与します。

入力: plan.md 他設計文書生成 → tasks.md
5

/speckit.analyze 整合性分析 推奨

spec / plan / tasks の横断的な矛盾・抜け漏れを非破壊でチェック。実装前の最終品質ゲートです。

入力: spec.md / plan.md / tasks.md出力 → 分析レポート（不整合の指摘）
6

/speckit.implement 実装の実行

tasks.md を上から順に実行し、実際のコードを生成。完了したタスクには [X] が付き、進捗が追跡されます。

入力: tasks.md生成 → 実際のソースコード / テスト
🧩 補助コマンド 上記の他に /speckit.checklist（任意の観点でチェックリスト生成）、 /speckit.taskstoissues（tasks.md を GitHub Issue 群に変換）などがあります。

5コマンド詳解

各コマンドの役割・入出力・使いどころを整理します。

コマンド役割主な出力
/speckit.constitutionプロジェクト原則（憲法）の作成・更新constitution.md
/speckit.specify自然言語の要望から要件仕様を作成spec.md
/speckit.clarify仕様の曖昧点を質問で明確化spec.md（更新）
/speckit.plan技術設計・実装計画を作成plan.md 他
/speckit.tasks計画を依存順タスクに分解tasks.md
/speckit.analyze成果物間の整合性を分析分析レポート
/speckit.checklist任意観点の品質チェックリスト生成checklists/*.md
/speckit.implementタスクを実行しコードを生成ソースコード
/speckit.taskstoissuesタスクを GitHub Issue に変換GitHub Issues

① constitution — プロジェクト憲法

最初に一度だけ実行し、プロジェクト全体で守るべき不変の原則を定めます。 テンプレートには「テストファースト（NON-NEGOTIABLE）」「シンプルさ（YAGNI）」など、原則の例が記載されています。 ここで定めた内容は plan フェーズで整合チェックの基準として使われます。

② specify — 「何を・なぜ」を固める

要望文から ユーザーストーリー（優先度付き）/ 受入シナリオ（Given-When-Then）/ 機能要件（FR）/ 成功条件 を生成します。 重要なのは 「実装の詳細（言語・FW・API）を書かない」こと。非技術者でも読める仕様を目指します。

⚠️ specify の流儀 曖昧な点は最大 3 個まで [NEEDS CLARIFICATION] マーカーで明示。 それ以外は「業界標準に基づく妥当な推測」で埋め、Assumptions セクションに記録します。 作成後は checklists/requirements.md で品質を自己検証します。

③ clarify — 曖昧さを潰す

spec.md を読み込み、判断に影響する曖昧点を選択肢付きの質問として提示。 回答は spec.md の「Clarifications」セクションに日付つきで追記されます。plan の前に実行するのが定石です。

④ plan — 「どう作るか」を設計する

ここで初めて技術選定を行います。技術スタック・アーキテクチャ・データモデル・API/UI コントラクトを決め、 必要に応じて research.md（調査）・data-model.md・contracts/ を生成します。 憲法に反する設計がないかもこの段階で検証されます。

⑤ tasks — 実行可能な単位に割る

計画を依存順に並んだタスクへ分解。このプロジェクトの実例では次のような構造になっています。

  • Phase 1: Setup — 共有インフラ（ページ雛形・カタログ登録）
  • Phase 2: Foundational — 全ストーリー共通の基盤（型・状態遷移ヘルパー）
  • Phase 3+: User Story 単位 — ストーリーごとにテスト → 実装をまとめる

各タスクには ID（T001…）、並列可能マーク [P]、対象ストーリー [US1]、 そして具体的なファイルパスが付きます。

⑥ analyze — 横断チェック

spec / plan / tasks を突き合わせ、「仕様にあるのにタスクが無い」「計画と仕様が矛盾」といった 整合性の問題を非破壊で洗い出す最終ゲートです。指摘を反映してから implement に進みます。

⑦ implement — 実装する

tasks.md を上から順に実行。Phase 単位のチェックポイントで動作確認しながら進み、 完了タスクは [X] でマークされて進捗がファイル上で可視化されます。

6プロンプト実例

実際の使い方は「Copilot Chat にスラッシュコマンド + 自然言語」を入力するだけ。各フェーズの具体的な指示例を示します。

セットアップ（CLI / 初回のみ）

# Spec Kit を取得して Copilot 連携で初期化
uvx --from git+https://github.com/github/spec-kit.git
specify init my-project --integration copilot

# 既存プロジェクトに導入する場合
specify init . --integration copilot

※ このリポジトリは導入済みのため、以降はチャットのスラッシュコマンドだけで進められます。

① 憲法を定める

/speckit.constitution このプロジェクトは学習用です。可読性を最優先し、
テストは Vitest で書く。過度な抽象化を避け、シンプルな実装を原則とする。

② 仕様を作る（何を・なぜ）

/speckit.specify 学習向けの簡易的アプリとしてシンプルな Todo アプリを作りたい。
タスクの作成・完了・削除ができる Web アプリケーションである。

③ 曖昧点を明確化

/speckit.clarify

→ AI が「削除時は確認を出す？」「並び順は？」等を選択肢付きで質問。答えると spec.md に反映される。

④ 技術設計（どう作るか）

/speckit.plan Next.js 16 App Router + React 19 + TypeScript で実装する。
状態はクライアント側のみ（ページセッション内保持）。UI は shadcn/ui を使う。

※ 公式例: 「.NET Aspire + Postgres + Blazor、REST API は projects/tasks/notifications…」のように技術を具体指定する。

⑤ タスク分解 → ⑥ 整合性分析 → ⑦ 実装

/speckit.tasks      # plan.md からタスク一覧を生成
/speckit.analyze    # spec/plan/tasks の矛盾をチェック（任意）
/speckit.implement  # tasks.md を順に実行してコード生成
✅ 進め方のコツ 各コマンドの後は必ず生成物を人間がレビューしてから次へ。 特に specify と plan の出力は、後工程の品質を大きく左右します。

7実プロジェクト例: シンプル Todo アプリ

このリポジトリの specs/001-simple-todo-app/ は、実際に Spec Kit で進めた成果物です。 各成果物の中身を抜粋で見てみましょう。

spec.md — ユーザーストーリーと受入シナリオ（抜粋）

### User Story 1 - タスクを作成する (Priority: P1)

学習者として、新しいタスクを入力して追加し、やるべきことを一覧に残したい。

**Acceptance Scenarios**:
1. Given タスク一覧が表示されている,
   When ユーザーがタスク名を入力して追加する,
   Then 新しい未完了タスクが一覧の末尾に表示される
2. Given 入力欄が空文字または空白のみ,
   When 追加を実行する,
   Then タスクは追加されず入力不備が明確に示される

spec.md — clarify で確定した仕様（抜粋）

## Clarifications — Session 2026-06-07
- Q: 「セッション内保持」の範囲は？
  → A: ページを開いている間のみ保持し、再読み込みで初期化する
- Q: タスク削除時の挙動は？
  → A: 削除時に確認を表示し、承認時のみ削除する
- Q: タスク名の文字数制約は？
  → A: 1〜100 文字に制限する

→ 曖昧だった点が「決定事項」として記録され、以降の設計・実装の根拠になる。

spec.md — 機能要件（FR / テスト可能な形）

FR-001: ユーザーはテキスト入力＋送信で新規タスクを作成できる
FR-002: 空・空白のみ・100 文字超の入力は拒否し、理由を伝える
FR-006: 削除は明示的な確認ステップの後にのみ実行できる
FR-009: タスク状態はページセッション中のみ保持し、再読込で初期化

tasks.md — 依存順タスク（抜粋）

## Phase 2: Foundational（全ストーリー共通基盤）
[X] T004 ドメイン型を定義  .../_components/todo-types.ts
[X] T005 [P] 純粋な状態遷移ヘルパー  .../_components/todo-state.ts
[X] T006 [P] 再利用可能な行コンポーネント  .../_components/TodoItem.tsx

## Phase 3: User Story 1（P1）🎯 MVP
[X] T009 [US1] 作成成功/検証失敗の unit テスト  .../TodoApp.test.tsx
[X] T010 [US1] 入力バリデーション実装  .../TodoApp.tsx
[X] T011 [US1] 作成順で追加するフロー  .../TodoApp.tsx
[X] 完了済みタスク [P] 並列実行可能 [US1] 対応ユーザーストーリー
📌 この例から分かること spec（FR-002 の入力制約）→ tasks（T010 のバリデーション実装）→ コード、という形で 要件が一本の線でトレース可能になっています。これが仕様駆動開発の最大の利点です。

8実践のコツ

🧱 spec では実装を書かない

言語・FW・API は plan で決める。spec は「何を・なぜ」に徹すると、技術が変わっても仕様が再利用できます。

🔍 clarify と analyze を省略しない

任意フェーズですが、手戻りを最も減らす品質ゲート。特に複雑な機能では効果絶大です。

👤 各フェーズ後に人間がレビュー

生成物をそのまま次に流さない。spec と plan のレビューが全体品質を決めます。

🗂️ 機能は specs/ 配下に独立管理

NNN-機能名/ 単位で成果物が完結。複数機能を並行して進めやすい構成です。

📜 憲法は最初に整える

テスト方針・品質基準を constitution に書くと、全フェーズで自動的に意識されます。

🔗 タスクを Issue 化して協業

/speckit.taskstoissues でチーム開発に展開。進捗を GitHub 上で追跡できます。
