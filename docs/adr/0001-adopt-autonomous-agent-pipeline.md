# ADR-0001: 自律型コーディングエージェント開発環境の採用

- Status: Accepted
- Date: 2026-06-10
- 関連: feature/orchestrator ブランチ, [docs/AUTONOMOUS_AGENT_ENVIRONMENT.html](../AUTONOMOUS_AGENT_ENVIRONMENT.html)

## Context（背景）

このプロジェクトでは、コーディングエージェントができる限り自律的に開発を進める環境を構築したい。
対象フローは (1) Issue 駆動の自動解決、(2) 提案駆動の機能開発、(3) 日次セキュリティ監査の3つで、
いずれも実装・テスト・PR 作成・レビュー・マージ・ADR 作成までを人間の関与を最小化して行う。
また、特定プロジェクトに依存しない汎用的な仕組みであることを重視する。

## Decision（決定）

**「単一のオーケストレーターエージェント」ではなく、「GitHub をイベントバス兼状態管理として使う
イベント駆動パイプライン」を採用する。**

- トリガーと品質ゲートは決定論的な仕組み（GitHub Actions・CI・ブランチ保護）が担う
- LLM（claude-code-action@v1 上の Claude Code）は各ステージ内の判断のみを担う
- 手順は `.claude/skills/`（resolve-issue / propose / security-audit / write-adr）に、
  プロジェクト固有知識は `CLAUDE.md` に分離する（汎用性の確保）
- 3つのフローはすべて「`claude:auto` ラベル付き Issue」という共通フォーマットに正規化して
  単一の実装パイプラインに合流させる
- 自律度は段階制とし、当面はレベル2（マージのみ人間）で運用する

## Alternatives（検討した代替案）

- **単一オーケストレーターエージェント方式**: 強力なエージェントがサブエージェントを指揮する構成。
  1セッション内の並列化には有効だが、システム全体の制御を LLM に委ねると判断の揺れ・コンテキスト肥大・
  失敗時のデバッグ困難・状態喪失の問題がある。2026年時点の実践知見では、信頼性は「賢さ」ではなく
  「構造」から生まれるとされるため却下。
- **完全自動マージを含む無人パイプライン**: Issue 本文は信頼できない入力であり、プロンプト
  インジェクションによる攻撃事例が多発している。claude-code-action の公式セキュリティガイドも
  自動マージを非推奨としているため却下。低リスクカテゴリに限定した条件付き自動マージ（レベル3）への
  拡大は、運用実績を見て将来判断する。
- **Claude Agent SDK による独自システム構築**: 柔軟性は最も高いが、claude-code-action と
  スキルの組み合わせで要件を満たせるため、保守コストに見合わず却下。

## Consequences（帰結）

- 各ステージが独立した GitHub Actions 実行になるため、失敗したステージだけ再実行できる
- 人間の承認行為は「Issue への `claude:auto` ラベル付け」と「PR のマージ」の2点に集約される
- ラベルゲートがプロンプトインジェクション対策を兼ねる（Issue 登録だけでは何も実行されない）
- API 利用料と GitHub Actions 実行時間が発生する（`--max-turns` とタイムアウトで上限を制御）
- スキルとワークフロー YAML のセットは他リポジトリへテンプレートとして転用できる
- フォローアップ: ラベル作成・ブランチ保護・`ANTHROPIC_API_KEY` シークレットの設定が必要
