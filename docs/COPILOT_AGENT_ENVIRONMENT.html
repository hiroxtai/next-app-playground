GitHub Copilot 版
自律パイプライン環境の構築

調査日: 2026年6月10日 ／ 対象リポジトリ: next-app-playground ／ 姉妹ドキュメント: Claude Code 版（設計の全体像）

結論（TL;DR）
Claude Code 版で設計したアーキテクチャ （イベント駆動・ラベルゲート・4層構成）はそのまま使える。 差し替えるのは実行層のみで、claude-code-action の役割を Copilot では Copilot coding agent（Issue を assign すると実装〜 draft PR 作成まで行うクラウドエージェント）と Copilot CLI（copilot -p による GitHub Actions からの無人実行)の2つが分担する。 このリポジトリは .github/copilot-instructions.md・.github/agents/・ .github/instructions/・.github/prompts/ が整備済みのため、共有定義層は大部分が構築済み。

1. Claude 版との対応関係

設計思想（決定論的なトリガーと品質ゲート、LLM はステージ内の判断のみ、共通フォーマットは「ラベル付き Issue」）は共通。実装要素の対応は次の通り。

役割Claude 版（構築済み）Copilot 版
Issue 自動解決claude-resolve-issue.yml + resolve-issue スキルCopilot coding agent（Issue を Copilot に assign）
メンション対応claude.yml（@claude）PR 上の @copilot コメント / Issue assign
PR 自動レビューclaude-review.ymlCopilot code review（設定/ルールセットで自動化。ワークフロー不要）

2. Copilot 版アーキテクチャ

Claude 版との最大の違いは、実装パイプラインの中身（ブランチ作成 → 実装 → テスト → PR 作成）を 自前のワークフロー + スキルで書く必要がなく、coding agent のネイティブ機能に委ねること。 自前で書くのは「ラベル → assign」の接続部分だけになる。

3. Copilot coding agent の有効化と環境準備

coding agent は GitHub 側のマネージド環境（GitHub Actions ベース、ファイアウォール付きサンドボックス）で動作し、 ブランチ作成 → 実装 → draft PR 作成までをネイティブで行う。

  1. 有効化: リポジトリの Settings → Copilot → Coding agent を有効にする（組織の場合はポリシー設定も確認）
  2. 環境準備: 依存関係を事前インストールする専用ワークフローを作成する。ファイル名とジョブ名は固定:
# .github/workflows/copilot-setup-steps.yml
name: "Copilot Setup Steps"
on: workflow_dispatch
jobs:
  copilot-setup-steps:   # ジョブ名はこの名前である必要がある
    runs-on: ubuntu-latest
    permissions:
      contents: read
    steps:
      - uses: actions/checkout@v6
      - uses: pnpm/action-setup@v5
        with:
          version: 9
      - uses: actions/setup-node@v6
        with:
          node-version: "22"
          cache: "pnpm"
      - run: pnpm install --frozen-lockfile
  • MCP サーバーの追加: Settings → Copilot → Coding agent の MCP configuration（JSON）で設定
  • ファイアウォール: 既定で外部通信が制限される。必要な許可ドメインは同設定画面で追加（無効化は非推奨）
  • カスタム指示: .github/copilot-instructions.md（整備済み）と AGENTS.md を coding agent も読む

4. ラベルゲート → API で Copilot に assign

2025年12月から GraphQL / REST API で Issue を Copilot に assign できるようになり、 Claude 版と同じラベルゲートが組める。assign 時に作業ブランチ・追加指示・使用するカスタムエージェントも指定できる。

重要な制約: デフォルトの GITHUB_TOKEN では Copilot を assign できない。 Copilot が有効なユーザーの PAT（user token）が必要。GitHub App / bot のトークンでは suggestedActors に copilot-swe-agent が返らない。 
# .github/workflows/copilot-resolve-issue.yml
name: Copilot Resolve Issue

# 人間が Issue に copilot:auto ラベルを付けると発火し、
# Copilot coding agent に assign する（以降は coding agent がネイティブで処理）
on:
  issues:
    types: [labeled]

jobs:
  assign:
    if: github.event.label.name == 'copilot:auto'
    runs-on: ubuntu-latest
    timeout-minutes: 5
    permissions:
      issues: write
    steps:
      - name: Assign issue to Copilot
        env:
          # Copilot 有効ユーザーの PAT（GITHUB_TOKEN では不可）
          GH_TOKEN: ${{ secrets.COPILOT_ASSIGN_PAT }}
          ISSUE_NUMBER: ${{ github.event.issue.number }}
          REPO: ${{ github.repository }}
        run: |
          # 1. Copilot エージェントの actor ID を取得
          AGENT_ID=$(gh api graphql \
            -H "GraphQL-Features: issues_copilot_assignment_api_support" \
            -f query='query($owner: String!, $name: String!) {
              repository(owner: $owner, name: $name) {
                suggestedActors(capabilities: [CAN_BE_ASSIGNED], first: 100) {
                  nodes { login ... on Bot { id } }
                }
              }
            }' \
            -F owner="${REPO%/*}" -F name="${REPO#*/}" \
            --jq '.data.repository.suggestedActors.nodes[] | select(.login == "copilot-swe-agent") | .id')

          # 2. Issue の node ID を取得
          ISSUE_ID=$(gh api "repos/${REPO}/issues/${ISSUE_NUMBER}" --jq '.node_id')

          # 3. Copilot を assign（既存の assignee は維持）
          gh api graphql \
            -H "GraphQL-Features: issues_copilot_assignment_api_support" \
            -f query='mutation($issueId: ID!, $agentId: ID!) {
              replaceActorsForAssignable(input: {assignableId: $issueId, actorIds: [$agentId]}) {
                assignable { ... on Issue { number } }
              }
            }' \
            -F issueId="$ISSUE_ID" -F agentId="$AGENT_ID"

assign 後の流れ（すべて coding agent のネイティブ機能）:

  1. Issue に 👀 リアクションが付き、セッション開始
  2. サンドボックス環境で調査 → 実装 → テスト（copilot-setup-steps.yml の環境で実行）
  3. draft PR を作成し、作業ログ（セッション）を PR にリンク
  4. PR コメントで @copilot に修正依頼すると追加作業する

5. PR 自動レビュー（Copilot code review）

ワークフロー不要。リポジトリ設定またはルールセットで「Automatic Copilot code review」を 有効化すれば、coding agent が作成した PR を含む全 PR に自動レビューが付く。

  • 設定場所: Settings → Rules → Rulesets（または Branch protection の自動レビュー設定）
  • レビュー観点のカスタマイズ: .github/instructions/*.instructions.md（整備済みの code-review-generic.instructions.md などが適用される）

6. Copilot CLI による定期監査・ADR 作成

cron 起動の無人実行は Copilot CLI のプログラマティックモード（copilot -p）で組む。 認証は COPILOT_GITHUB_TOKEN 環境変数に 「Copilot Requests」権限付きの fine-grained PAT を渡す。

# .github/workflows/copilot-security-scan.yml
name: Copilot Security Scan

# 毎日 JST 9:00 にセキュリティ監査を実行し、問題があれば Issue を起票する
# このジョブはコードを変更しない（contents: read のみ）
on:
  schedule:
    - cron: "0 0 * * *"
  workflow_dispatch:

jobs:
  audit:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    permissions:
      contents: read
      issues: write
    steps:
      - uses: actions/checkout@v6
      - uses: pnpm/action-setup@v5
        with:
          version: 9
      - uses: actions/setup-node@v6
        with:
          node-version: "22"
          cache: "pnpm"
      - name: Install Copilot CLI
        run: npm install -g @github/copilot
      - name: Run security audit
        env:
          COPILOT_GITHUB_TOKEN: ${{ secrets.COPILOT_CLI_PAT }}
          GH_TOKEN: ${{ github.token }}
        run: |
          copilot -p "$(cat .github/prompts/security-audit.prompt.md)" \
            --allow-tool='shell(pnpm audit:*)' \
            --allow-tool='shell(pnpm outdated:*)' \
            --allow-tool='shell(gh issue:*)' \
            --allow-tool='shell(gh pr list:*)' \
            --allow-tool=read \
            --no-ask-user
CI では --allow-all を絶対に使わない（公式ドキュメントも明記）。 必要なツールだけを --allow-tool で列挙する。監査ジョブは読み取り系コマンドと Issue 操作のみに制限し、コード変更はラベルゲート経由のパイプラインに委ねる。

ADR 自動作成も同じパターンで組める。PR の merged イベントをトリガーに、 write-adr 相当のプロンプトを Copilot CLI で実行する（または「ADR を書く Issue」を起票して coding agent に assign する方式でもよい）。

7. スキル（手順書）の移植

Claude 版の .claude/skills/ 4点は次のように移植する。内容（手順・セキュリティ注意・完了条件）はほぼそのまま流用できる。

Claude 版スキルCopilot 版の置き場所備考

8. セキュリティと PAT の管理

自律度の段階制（レベル1: ラベルゲート → レベル2: マージのみ人間 → レベル3: 条件付き自動マージ）の考え方は Claude 版と共通。Copilot 固有の事情は次の通り。

  • 人間ゲートが製品仕様として組み込み: coding agent は必ず draft PR を作る。さらに 「coding agent の PR はトリガーした本人の approve だけでは CI が自動実行されない」等の保護が既定で入っており、 自律度レベル2が事実上強制される
  • PAT が2種類必要:
    シークレット名用途要件
    COPILOT_ASSIGN_PATIssue を Copilot に assignCopilot 有効ユーザーの user token（Issues: write）
    COPILOT_CLI_PATCopilot CLI の認証fine-grained PAT + Copilot Requests 権限
    どちらも個人アカウントに紐づくため、有効期限の管理と権限の最小化に注意する
  • プロンプトインジェクション対策: ラベルゲート（人間がラベルを付けることが承認行為）は Copilot 版でも同様に機能する。coding agent 側にもファイアウォール・サンドボックスの防御層がある
  • 課金: coding agent はタスクごとに premium requests + GitHub Actions 分を消費。 CLI も premium requests を消費する。timeout-minutes と実行頻度で上限を制御する

9. セットアップ手順チェックリスト

  1. リポジトリの Settings → Copilot → Coding agent を有効化
  2. .github/workflows/copilot-setup-steps.yml を作成（pnpm 環境の事前構築）
  3. ラベルを作成: gh label create "copilot:auto" --description "Copilot による自動処理を承認" --color 1D9E75
  4. PAT を2種類発行し、リポジトリシークレットに登録（COPILOT_ASSIGN_PAT / COPILOT_CLI_PAT）
  5. copilot-resolve-issue.yml（assign ワークフロー）を作成
  6. ルールセットで Automatic Copilot code review を有効化
  7. .github/prompts/ に propose / security-audit / write-adr を移植
  8. copilot-security-scan.yml（cron）と ADR ワークフローを作成
  9. ブランチ保護（main 直 push 禁止 + required checks）は Claude 版と共通
  10. 試運転: まず Issue を1件手動で Copilot に assign し、draft PR の品質を確認してからラベル運用を開始

10. Claude 版との違い・併用

観点Claude Code 版Copilot 版
実装パイプラインの構築スキル + ワークフローを自前で書く（柔軟）coding agent のネイティブ機能（構築が少ない）
人間ゲート設計で担保（ラベル + ブランチ保護）製品仕様で強制（draft PR 等）
手順のカスタマイズ性高い（スキルで細かく制御）instructions / prompts 経由（coding agent の内部手順は変更不可）
認証ANTHROPIC_API_KEY + GitHub App個人 PAT 2種類（管理に注意）
課金API トークン従量 + Actionspremium requests + Actions

併用も可能。ラベルを claude:auto / copilot:auto で分ければ同居でき、 「実装は Copilot coding agent、レビューは Claude」のようなクロスレビュー構成も組める。 どちらか一方に寄せる場合も、共有定義層（instructions・ADR・ブランチ保護）とラベルゲートの設計は共通資産として残る。
